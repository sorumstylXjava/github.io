What is AxManager and How to Use It

AxManager is a non-root module manager running on Axeron — its own independent privileged service.

AxManager is a non-root module manager for Android. It runs on top of the Axeron privileged service — its own independent server similar in concept to Shizuku, but built specifically for the AxManager ecosystem.

What is Axeron?

Axeron is AxManager's own privileged service, comparable to Shizuku but independent. Like Shizuku, it connects via ADB Wireless Debugging to gain elevated permissions — but it is a separate service built specifically for the AxManager module ecosystem, not the Shizuku API.

How Axeron Connects

The connection method is the same as Shizuku — ADB Wireless Debugging. Enable Wireless Debugging in Developer Options, then pair Axeron through AxManager the same way you would pair Shizuku. No root required.

Requirements

  • Android 11 or higher
  • AxManager app installed
  • Developer Options and Wireless Debugging enabled
  • Modules that declare axeronPlugin=13000 in their metadata

How to Get Started

1. Enable Wireless Debugging in Developer Options.

2. Open AxManager and start the Axeron service — pair it via Wireless Debugging when prompted.

3. Download a compatible module — any module on this site tagged with AxManager supports it.

4. In AxManager go to Modules → Install from Storage, select the zip file, and wait for the flash to complete.

5. Module is now active. No reboot required for most non-root modules.

What AxManager Modules Can Do

  • Apply setprop tweaks — system properties affecting performance and behavior
  • Disable system tracing and profiling — reduces background CPU overhead
  • Modify SurfaceFlinger parameters for smoother rendering
  • Apply display and animation tweaks
  • Adjust memory management parameters within ADB-accessible limits

Is AxManager Compatible With Rooted Devices?

Yes. Modules tagged with both root and AxManager work on both setups. On a rooted device the root path is used for deeper access. On non-rooted devices with Axeron, the AxManager path is used. The module detects the environment at runtime.

AxManager vs Magisk — A Direct Comparison

Magisk and AxManager solve similar problems but for different audiences. Magisk is for rooted devices and can modify the system at a deep level including kernel parameters and system partitions. AxManager is for non-rooted devices and works within the limits of what ADB permissions allow through the Axeron service.

The practical difference: if a tweak requires writing to /system or modifying kernel parameters, it needs Magisk. If a tweak can be done through system APIs or setprop commands accessible via ADB, AxManager can handle it. Many common performance tweaks fall into the second category.

What is axeronPlugin=13000?

This is a metadata flag in a module's configuration that declares compatibility with the Axeron service. When AxManager installs a module, it checks for this flag to know how to handle the module's scripts. Modules without this flag are treated as Magisk-only and will not run correctly in AxManager's non-root environment.

Developers who build modules for both environments include this flag along with conditional logic in their scripts — checking whether root is available and falling back to ADB-compatible commands if not. This is how modules on this site work when tagged with both root and AxManager.

Axeron Service Lifecycle

Like Shizuku, the Axeron service stops when the device reboots. You need to restart it through AxManager after each reboot. This is a fundamental limitation of ADB-based privilege escalation on non-rooted Android. The reconnection process is quick — open AxManager, start the Axeron service, and it re-establishes the Wireless Debugging connection automatically if pairing was done previously.

Debugging Module Issues

If a module does not seem to be applying its tweaks after installation, the most common causes are: the Axeron service not running, the module requiring root-level access that Axeron cannot provide, or the module's scripts using commands not available in ADB shell. AxManager's logs section shows output from module scripts, which helps identify exactly where a failure occurs.

Building Your Own AxManager Module

AxManager modules follow a similar structure to Magisk modules — a zip file containing a module.prop for metadata and shell scripts for the actual operations. The key difference is that scripts must only use commands available in ADB shell. Adding axeronPlugin=13000 to module.prop declares Axeron compatibility. This makes the barrier to entry relatively low for anyone familiar with Android shell scripting.
